## 그룹 A: Inversion (방향 뒤집기)

### L1: Direction Reversal
```
다음 인사이트를 받아서 인과/주체의 방향을 정확히 반대로 한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 인사이트에서 주체(누가), 객체(누구를/무엇을), 작용(어떻게)을 식별
2. 주체↔객체 교환, 작용 방향 반전
3. 반전된 구조가 작동하는 시스템을 묘사

예: "검색엔진이 사용자에게 콘텐츠를 추천" → "사용자가 검색엔진을 학습시키고, 그 학습값이 거래되는 시장"

조건:
- 단순 부정(not X)이 아닌 *구조적 반전*
- 결과 시스템이 일관된 stakeholder/cashflow를 가질 것
- "Lens trace": 무엇이 무엇으로 반전되었는지 명시
```

### L2: Stakeholder Inversion
```
다음 인사이트의 권력 관계를 뒤집은 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 갑(권력자)과 을(피권력자) 식별
2. 두 역할을 교환
3. 교환된 권력 관계에서 유지되어야 할 기능을 새 시스템으로 구현

예: "기업이 신용평가사를 평가" → "신용평가사가 기업의 평가 권한을 입찰받음 / 평가받는 자가 평가자를 선택하는 역경매"

조건:
- 권력 관계가 명시적으로 뒤집힐 것
- 새 권력자의 인센티브 구조 명시
- 기존 권력자가 어디로 가는지 설명
```

### L3: Cost Inversion
```
다음 인사이트의 비용 구조를 뒤집은 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 누가 비용을 지불하고 누가 수익을 가져가는지 식별
2. 비용↔수익 흐름을 뒤집기
3. 뒤집힌 흐름이 작동하는 메커니즘 설계 (광고 모델, 보조금, 보험 등)

예: "데이터센터가 전력 비용을 지불" → "데이터센터가 전력망에 *지불받는* (수요 반응으로 안정성 제공)"

조건:
- 결과 시스템에서 누가 새 지불자, 새 수익자인지 명시
- 비용 흐름의 *방향*이 명확히 반전
```

### L4: Privacy Inversion
```
다음 인사이트의 정보 공개/비공개 구조를 뒤집어라.

인사이트: {insight.statement}

변형 절차:
1. 현재 무엇이 공개되고 무엇이 비공개인지 식별
2. 둘을 교환
3. 새 정보 흐름에서 가능한 시스템 설계

예: "은행 잔고는 비공개, 직장은 공개" → "은행 잔고 공개, 직장은 비공개. → 부의 공개기반 신뢰 네트워크 (마사이족 cattle wealth와 유사)"

조건:
- 결과 시스템에서 정보 비대칭이 어떻게 변하는지 명시
- 새 시스템의 attack surface 식별
```

### L5: Ownership Inversion
```
다음 인사이트의 소유 구조를 뒤집어라.

인사이트: {insight.statement}

변형 절차:
1. 현재 누가 무엇을 소유하는지 식별 (자산, 데이터, 권리)
2. 소유권을 다른 stakeholder에게 이전
3. 새 소유 구조에서 작동하는 거래·임대·라이선스 시스템 설계

예: "플랫폼이 사용자 데이터 소유" → "사용자가 데이터 소유, 플랫폼이 데이터 사용권을 임대 (data dividend)"

조건:
- 소유권 이전이 *법적·기술적으로* 어떻게 구현되는지 명시
- 기존 소유자가 어떻게 보상되는지 (또는 안 되는지)
```

### L6: Trust Model Inversion
```
다음 인사이트의 신뢰 모델을 뒤집어라.

인사이트: {insight.statement}

변형 절차:
1. 현재 누가 신뢰의 *근원*인지 식별 (정부, 기업, 알고리즘, 합의)
2. 신뢰 위계를 뒤집기 (중앙↔분산, 인증자↔피인증자)
3. 새 신뢰 모델에서 작동하는 검증 시스템 설계

예: "정부가 신원 인증" → "시민들이 정부 기관의 신뢰도를 인증 (정부 reputation score)"

조건:
- 검증 메커니즘이 *어떻게* 작동하는지 기술적으로 명시
- Trust assumption이 어디로 옮겨갔는지 설명
```

### L7: Failure-as-Feature
```
다음 인사이트에서 가장 큰 결함·실패·문제를 *핵심 기능*으로 전환한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 인사이트에서 명시되거나 함의된 결함·실패 모드 식별
2. 그 실패를 *해결*하는 게 아니라 *활용*하는 시스템 설계
3. 실패가 이 시스템에서 어떻게 가치를 생산하는지 명시

예: "AI hallucination = 버그" → "AI hallucination = 통제된 창의성 발생기 (소설·아이디어 생성용 hallucination dial)"
예: "IoT가 PQC 마이그레이션 불가" → "마이그레이션 불가성을 *기반*으로 한 보험 시장"

조건:
- 실패가 기능으로 전환되는 *메커니즘*이 명시적
- 결함을 단순히 무시·은폐하는 게 아니라 *활용*
- 결과 시스템에서 결함이 더 클수록 가치가 더 커지는 구조 우선
```

### L8: Side Effect Mining
```
다음 인사이트의 *부산물·외부효과·노이즈*를 본질적 가치원으로 만든 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 인사이트의 주제에서 발생하는 부산물·외부효과·폐기물·노이즈 식별
2. 그 부산물 자체가 핵심 가치가 되는 시스템 설계
3. 원래 주제는 부수적 역할로 재배치

예: "데이터센터가 열을 폐기" → "열이 본질, 데이터 처리는 부산물 (지역난방 일차 + 무료 컴퓨팅 이차)"
예: "AI 학습 부산물 = 임베딩" → "임베딩 자체가 거래되는 시장, 모델은 임베딩 생산 도구"

조건:
- 부산물이 *가치 척도*가 되어야 함
- 원래 주제와 부산물의 우선순위가 *실제로* 뒤바뀜
```

## 그룹 B: Shift (차원 이동)

### L9: Scale Shift
```
다음 인사이트를 1000배 큰 규모 또는 1000배 작은 규모로 이동시켜 작동하는 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 인사이트의 작동 규모 식별 (개인, 도시, 국가, 행성, 분자, 세포 등)
2. ±3 자릿수 규모 이동
3. 새 규모에서 가능해지는 시스템 설계 (다른 규모에선 불가능)

예: "도시 단위 에너지 거래" → 가전 단위 (마이크로 그리드 거래) / 행성 단위 (우주 태양광 PPA)
예: "개인 건강 추적" → 분자 단위 (단백질 수준 모니터링) / 인구 단위 (역학 실시간 추적)

조건:
- 새 규모에서 *물리적·경제적·사회적*으로 가능한지 검토
- 다른 규모에서는 의미없는 시스템이어야 함 (규모 특이성)
```

### L10: Time Shift
```
다음 인사이트가 50년 전 또는 100년 후에 있었다면 어떤 시스템이었을지 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 인사이트의 핵심 문제가 다른 시대에도 존재했는지/할지 식별
2. 그 시대의 기술·제도·문화로 *동등한* 시스템 설계
3. 그 시대 솔루션을 *현재로 다시 가져와* 가능한가 검토

예: "AI 모델 검증" → 100년 후: 자기 검증 AI (현재 인간 검증) / 50년 전: 전문가 위원회 + 출판 동료심사 → 현재 부활?

조건:
- 시대 이동이 *기술적 가능성*에 맞게 조정
- 시대간 이식 가능성 (현재로 가져올 수 있는 요소) 명시
- 100년 후 시나리오는 SF가 아닌 *합리적 외삽*
```

### L11: Domain Transplant
```
다음 인사이트의 구조를 *완전히 다른* 분야로 이식한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 인사이트의 추상 구조 추출 (행위자, 흐름, 제약, 인센티브)
2. 멀리 떨어진 분야 식별 (의료↔농업, 군사↔교육, 금융↔스포츠, 우주↔심해 등)
3. 추상 구조를 새 분야의 용어·메커니즘으로 재구현

예: "면역계의 분산 방어" → 사이버보안 시스템
예: "도시 교통 신호" → 비행기 routing 협상
예: "결혼시장의 매칭" → 우주 발사 슬롯 매칭

조건:
- 분야 거리가 멀어야 함 (인접 분야 X)
- 추상 구조의 핵심 속성이 보존되어야 함
- 새 분야에서 *진짜로* 작동 가능한지 검증
```

### L12: Granularity Shift
```
다음 인사이트의 처리 단위(granularity)를 극단적으로 변경한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 처리 단위 식별 (월/일/시간/분/초, 묶음/개별, 사람/팀/조직)
2. 한 자릿수 이상 더 잘게 or 더 굵게 변경
3. 새 단위에서 가능해지는 시스템 설계

예: "월 단위 클라우드 청구" → 함수 호출 단위 청구 (이미 Lambda 부분 구현) → 모델 token 단위 → CPU cycle 단위
예: "직무 단위 고용" → 작업 단위 (Mechanical Turk) → 5초 단위 (마이크로 노동)

조건:
- 새 granularity에서 측정/거래 *비용*이 감당 가능해야 함
- 다른 단위에선 의미 없는 시스템이어야 함
```

### L13: Frequency Shift
```
다음 인사이트의 발생 빈도를 극단적으로 변경한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 발생 빈도 식별 (실시간/시간당/일별/연간/평생 1회)
2. ±3 단계 이상 빈도 변경
3. 새 빈도에서 가능해지는 시스템 설계

예: "주식 거래 = 밀리초 단위" → 1년 1회 거래 (강제 long-term holding 시장)
예: "투표 = 4년 1회" → 실시간 (Liquid Democracy)
예: "결혼 = 평생 1회 (관습)" → 5년 갱신 계약

조건:
- 빈도 변경이 *제도·기술적으로* 구현 가능
- 새 빈도가 만들어내는 행동 변화 명시
```

### L14: Medium/Material Swap
```
다음 인사이트의 구현 매체를 완전히 다른 매체로 교체한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 구현 매체 식별 (디지털, 종이, 금속, 생물, 화학, 광자 등)
2. 멀리 떨어진 매체로 교체
3. 새 매체에서 작동하는 시스템 설계 (단순 이식 X, 매체 특성 활용)

예: "디지털 데이터 저장" → DNA 저장 / 광결정 / 단백질 폴딩
예: "이메일 통신" → 페로몬 통신 (생체) / 빛 깜빡임 (Li-Fi) / 음향 통신 (수중)
예: "암호화" → 화학적 자물쇠 (특정 효소로만 풀림)

조건:
- 새 매체의 *고유 속성*을 활용 (단순 이식 X)
- 매체 특성이 시스템에 새 기능을 부여
- 물리적으로 구현 가능 (SF 아님)
```

## 그룹 C: Constraint (제약 조작)

### L15: Constraint Removal
```
다음 인사이트가 속한 분야에서 *모두가 당연시*하는 제약을 식별하고 제거한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 분야의 암묵적 제약 5개 나열 (운전면허, 신원, 국경, 저작권, 정해진 working hours 등)
2. 그 중 *제거 시 가장 큰 변화*를 일으킬 제약 1개 선택
3. 제거된 후의 시스템 설계

예: "신용 평가에 직업 이력 필수" → 직업 이력 없이 평가 (대안 신호 시스템)
예: "비행기 = 정해진 항공로" → 항공로 없는 동적 경로 (이미 부분 구현)
예: "약물 = FDA 승인 필요" → 사용자 동의 기반 직접 접근 (right-to-try 확장)

조건:
- 제거 대상이 *진짜로 당연시*되는 것이어야 함 (사소한 제약 X)
- 제거 후 발생할 부작용 명시
- 부작용을 관리할 *대체 메커니즘* 포함
```

### L16: Constraint Addition
```
다음 인사이트의 시스템에 극단적 제약을 추가하여 단순화/집중을 강제하는 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 자유로운 측면 식별
2. 극단적 제약 부과 (1가지만, X시간만, X자원만 등)
3. 제약 하에 작동하는 *집중된* 시스템 설계

예: "휴대폰 = 키패드+화면 자유" → 키패드 제거 (iPhone)
예: "SNS = 다양한 콘텐츠" → 텍스트 280자만 (Twitter)
예: "메뉴 = 다양" → 메뉴 1개 식당
예: "AI prompt = 자유 토큰" → 단어 10개 제한 → 강제 압축적 의미 표현

조건:
- 제약이 *극단적*이어야 함 (소소한 제약 X)
- 제약으로 인한 *집중·단순화*가 핵심 가치
- 잃는 것보다 얻는 것이 명확
```

### L17: Constraint Substitute
```
다음 인사이트의 *핵심 제약*을 다른 종류의 제약으로 교체한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 핵심 제약 식별 (비용, 시간, 공간, 신뢰, 자원, 법규 등)
2. 다른 종류의 제약으로 교체
3. 새 제약 하의 시스템 설계

예: "전기차 = 배터리 용량 제약(거리)" → 시간 제약(충전 시간이 고정, 거리 자유) → 배터리 swap 시스템
예: "동영상 = 대역폭 제약" → 화질 제약 → 적응 스트리밍 / 신뢰 제약 → 영상 진위 검증 시장
예: "고용 = 학력 제약" → 신뢰 제약 → 평판 기반 고용

조건:
- 제약의 *종류*가 바뀌어야 함 (양적 조정 X)
- 새 제약이 합리적 (사용자가 수용 가능)
- 제약 교체로 인한 시장 재편 명시
```

## 그룹 D: Composition (합성·분해)

### L18: Atomization
```
다음 인사이트의 처리/거래/소유 단위를 극단적으로 *atomize*한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 가장 작은 단위 식별
2. 그것을 더 작게 쪼갠다 (소수점 이하, 가상 단위, 시간 슬라이스)
3. 잘게 쪼개진 단위가 거래·결합 가능한 시스템 설계

예: "주식 = 1주 단위" → 0.0001주 (이미 구현) → 의결권만 분리 거래
예: "기사 = 단위" → 문단 단위 / 문장 단위 / 데이터 포인트 단위
예: "고용 = 시간 단위" → 1분 단위 / 작업 단위 / 의사결정 1회 단위

조건:
- Atomization으로 *새로운 시장*이 발생
- 작은 단위의 거래·관리 비용이 감당 가능 (기술 발전 가정)
- 큰 단위로 묶는 *역방향* 메커니즘도 함께 설계 (재구성 자유도)
```

### L19: Aggregation
```
다음 인사이트의 분산된 작은 것들을 큰 단위로 묶어서 새 가치를 만드는 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 분야의 분산된 작은 단위들 식별 (개별 사용자, 개별 디바이스, 개별 거래)
2. 묶음으로 발생하는 가치 식별 (네트워크 효과, 협상력, 분포 정보)
3. Aggregator로서의 시스템 설계

예: "개별 신용카드 멤버십" → 통합 멤버십 / 통합 포인트 거래소
예: "개별 환자 데이터" → 인구 기반 분석 플랫폼 (협동조합 모델)
예: "개별 GPU" → 분산 GPU 풀 (Render, Akash 등)

조건:
- 묶음 자체가 *비선형 가치*를 발생
- 개별 단위들의 인센티브 정렬 메커니즘 명시
- Aggregator의 leverage point 명시
```

### L20: Multi-Lens Stack (가장 강한 변형)
```
다음 인사이트에 2-3개 렌즈를 순차 적용한 깊은 변형 시스템을 설계하라.

인사이트: {insight.statement}
Lens stack: {lens_1} → {lens_2} → ({lens_3})

변형 절차:
1. 첫 번째 렌즈 적용 → 중간 결과 1
2. 중간 결과 1에 두 번째 렌즈 적용 → 중간 결과 2
3. (선택) 세 번째 렌즈 적용 → 최종
4. 각 단계의 *변형 흔적*을 명시

조합 규칙:
- 같은 그룹 내 렌즈 2개 사용 금지 (그룹 A–E 간 조합만)
- 효과적인 페어 예:
  * L7(Failure-as-Feature) → L11(Domain Transplant): 한 분야의 결함을 다른 분야로 이식
  * L1(Direction Reversal) → L9(Scale Shift): 반전을 다른 규모에서 구현
  * L15(Constraint Removal) → L19(Aggregation): 제약 제거 후 통합
  * L8(Side Effect Mining) → L17(Constraint Substitute): 부산물을 새 자원 제약으로
  * L23(Counterparty Creation) → L19(Aggregation): 반대편을 집계해 시장 형성  [E×D]
  * L21(Recursion) → L11(Domain Transplant): 자기참조 구조를 다른 분야로 이식  [E×B]
  * L25(Stock↔Flow) → L9(Scale Shift): 흐름 전환을 다른 규모에서 구현  [E×B]
  * L24(Boundary Redraw) → L3(Cost Inversion): 경계 재설정 후 비용 흐름 반전  [E×A]

예 (3-stack):
  L11(Domain Transplant: 의료→농업) → L7(Failure-as-Feature: 잡초를 자원으로) → L19(Aggregation: 지역 잡초 자원 풀)
  → "지역 잡초 협동조합 식물성 의약품 원료 생산 시스템"

조건:
- 각 렌즈 적용 단계가 *서로 다른 차원*에서 변형
- 최종 결과가 단일 렌즈로 도달할 수 없는 새 시스템
- Stack trace 명시 (어떤 렌즈가 무엇을 변형했는지)
```

## 그룹 E: Restructuring (구조 재배치)

> v1.6 신설. A(속성 반전)·B(좌표 이동)·C(제약 조작)·D(단위 합성/분해)가 *요소 자체*를 바꾼다면,
> 그룹 E는 **구성요소 간의 구조적 관계**(자기참조·순서·반대편·경계·상태형태)를 재배치한다.
> CIX-20260607-001 backtest 통과(§lens_catalog 상단 주석).

### L21: Recursion / Self-Application
```
다음 인사이트의 핵심 메커니즘을 그 자신 또는 상위 메타-레이어에 다시 적용한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 인사이트의 핵심 메커니즘(작용)을 식별
2. 그 메커니즘을 자기 자신, 또는 그것을 측정/관리/거래하는 메타 시스템에 적용
3. 자기참조가 안정적 고정점(fixed point)에 수렴하는 시스템 설계

예: "AI가 코드를 검증" → "AI가 자신의 검증기를 검증·거래하는 메타 시장"
예: "센서가 냉각을 측정" → "측정 장치가 자신의 측정 비용을 자기 계량·과금"

조건:
- 무한루프가 아닌 *안정적 고정점/수렴* 구조일 것
- 메타-레이어가 원본보다 더 큰 가치를 포착하는 이유 명시
- 단순 반복이 아닌 *자기적용으로 새 가치*가 발생
```

### L22: Sequence Reorder
```
다음 인사이트가 함의하는 프로세스의 단계 순서·선후관계를 재배열한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 프로세스의 단계 순서 식별 (A → B → C)
2. 순서를 재배열 (후불, 사후검증, just-in-time, 사전 provisioning 등)
3. 재배열로 인센티브·리스크·신뢰가 어떻게 이동하는지 설계

예: "결제 후 배송" → "배송 후 결제 (후불 신뢰 시장)"
예: "siting 후 water 확보" → "water 확보 후 siting (water-first 허가)"
예: "생성 후 검증" → "검증 escrow 후 생성 (pre-committed gate)"

조건:
- 단순 방향반전(L1)이 아닌 *단계 순서/타이밍*의 재배열
- 재배열을 가능케 하는 신뢰·에스크로·예측 메커니즘 명시
- 순서 변경이 만드는 인센티브 이동을 명시
```

### L23: Counterparty Creation
```
다음 인사이트의 단면(one-sided) 활동에 빠져 있는 반대편 시장/역할을 신설한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 단면으로만 존재하는 활동 식별 (예측, 생성, 주장, 공급)
2. 그것에 반대로 베팅·검증·공매도·bounty하는 반대편 당사자 설계
3. 양측 인센티브가 정렬되어 진실/품질을 강제하는 시장 구조 설계

예: "예측만 존재" → "예측 실패에 베팅하는 공매도 시장"
예: "생성기가 자유 생성" → "오용을 잡아 보상받는 bounty 반대편"
예: "공급자가 주장하는 용량" → "과장 주장에 short 치는 반대편"

조건:
- 반대편이 *경제적으로 자생*하는 구조 (보상 출처 명시)
- 반대편의 존재가 원래 활동의 정직성/품질을 강제
- 합성·분해(그룹 D)가 아닌 *반대 역할의 신설*
```

### L24: Boundary Redraw
```
다음 인사이트 시스템의 내부/외부 경계(무엇이 안이고 무엇이 밖인가)를 다시 그은 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 시스템 경계 식별 (무엇을 내부화하고 무엇을 외부에 두는가)
2. 경계 재설정 (외부효과 내부화, 번들↔언번들, 수직통합↔분해)
3. 새 경계에서 비용·책임·통제가 어떻게 재배치되는지 설계

예: "데이터센터 ↔ 외부 유역(commons)" → "유역을 내부화한 단일 수문 단위"
예: "검증 = 외부 게이트" → "검증을 합성 하드웨어 내부 막(membrane)으로"

조건:
- 권력반전(L2)이 아닌 *범위/경계*의 재설정
- 내부화/외부화로 어떤 외부효과가 *가격화*되는지 명시
- 경계 이동으로 누가 새 비용·책임을 지는지 명시
```

### L25: Stock ↔ Flow
```
다음 인사이트의 정적 재고·소유(stock)를 연속 흐름·스트림(flow)으로, 또는 그 반대로 전환한 시스템을 설계하라.

인사이트: {insight.statement}

변형 절차:
1. 현재 stock(소유·재고·1회성 자격)인지 flow(흐름·구독·스트림)인지 식별
2. 반대 형태로 전환
3. 전환된 형태에서 연속 가격책정·계량이 가능한 시스템 설계

예: "물 권리(소유) → 리터-초 단위 흐름 구독"
예: "1회 인증서(stock) → 사용마다 소모되는 신뢰 스트림"
예: "소유 컴퓨트(stock) → 비용지표 연동 컴퓨트 스트림"

조건:
- 비용반전(L3)이 아닌 *형태(stock/flow)* 의 전환
- 연속 계량·과금이 기술적으로 감당 가능
- 전환이 만드는 행동 변화(소유→접근) 명시
```
